# 梅雨锋新流水线技术架构与 Runner 升级设计文档

## 1. 目标

本设计文档定义两个连续交付物：

1. 一份面向维护者的“技术架构与旧工程映射”文档
2. 一次面向当前已验证链路的 `runner` 升级

这两个交付物服务同一个目标：把当前新流水线从“模块已经具备、需要人工理解和拼接”推进到“模块边界清楚、旧新关系清楚、最小已验证链路可以由统一入口串联执行”。

本次设计不追求一步到位的全自动科研总控器，也不追求 front1 / front2 / CRA40 / ERA5 / 多变量的统一泛化框架。重点是先把当前已经真实验证过的链路正式产品化。

## 2. 当前状态

### 2.1 已完成部分

当前项目已经完成了以下基础工作：

- 旧工程主要脚本已在尽量少修改逻辑的前提下完成项目内路径迁移与冒烟验证
- 新流水线的核心步骤模块已经建立并通过测试
- `masks -> geometry -> profiles -> subareas -> statistics` 这条主链路已经用真实案例完成验证
- 中文路径兼容层 `nc_compat.py` 已经形成项目级基础设施
- 第一份使用者文档“快速使用指南”已经完成

### 2.2 当前缺口

虽然当前模块已经可用，但仍存在两个结构性缺口：

#### 缺口一：维护视角下的新旧关系还不够清楚

当前已经有：

- 分层设计文档
- legacy 冒烟基线文档
- 快速使用指南

但还缺一份专门回答“新模块承接了旧工程哪些逻辑、哪些已经迁移、哪些还留在 legacy 侧”的文档。

#### 缺口二：`runner` 还是基础入口

当前 [`pipeline/runner.py`](/H:/邢台观测站/CWR_project/meiyu_new/pipeline/runner.py) 只做：

- 读取配置
- 创建输出目录
- 执行 `inventory`
- 执行 `masks`

它还没有真正串联：

- `geometry`
- `profiles`
- `subareas`
- `statistics`

因此使用者虽然可以按步骤跑通，但还没有一个统一、稳定、面向当前最小案例的串联入口。

## 3. 设计范围

本次设计范围严格限制在以下两项：

### 3.1 交付一：技术架构与旧工程映射文档

新增一份维护文档，目标不是教人“怎么跑”，而是回答：

- 当前新流水线各模块边界是什么
- 每个模块对应旧工程中的哪些脚本或逻辑段
- 哪些能力已经迁移完成
- 哪些能力仍停留在 legacy 侧
- 后续如果扩展 front1、ERA5、更多变量，应该从哪一层接入

### 3.2 交付二：`runner` 轻量升级

升级 `runner`，使其能够串联当前唯一已经完成真实验证的链路：

- 数据集：`CRA40`
- 锋面：`front2`
- 时次：`2017-06-22T18`
- 变量：先聚焦已验证示例中的 `rh`

升级后的 `runner` 目标是“可靠串联器”，不是“过早泛化的全自动总控器”。

## 4. 不在本次范围内的内容

为了保证本次交付稳定，下列内容明确不纳入本次范围：

- front1 的统一自动串联入口
- ERA5 剖面链路的统一自动串联入口
- 多变量批处理型 profile 总控
- 诊断图自动生成总控
- 自动落盘完整几何文件、完整 profile 数据文件、完整子区域文件体系
- 重新设计或替换现有科研算法细节

这些都属于后续扩展阶段，不适合和当前“已验证链路正式产品化”混在同一轮中完成。

## 5. 可选方案比较

### 5.1 方案 A：文档先行 + 轻量串联 Runner

先写“技术架构与旧工程映射”，再让 `runner` 串联当前已验证链路，并返回结构化执行摘要。

优点：

- 最稳妥
- 范围清晰
- 不会把未验证能力绑进统一入口
- 符合当前项目成熟度

缺点：

- `runner` 仍然不是通用总控器
- 输出主要还是摘要和少量结果，不是完整全落盘流程

### 5.2 方案 B：文档 + Runner + 更多文件落盘

在方案 A 的基础上，再让 `runner` 自动保存更多中间文件。

优点：

- 更接近完整研究流水线
- 对部分使用者更直观

缺点：

- 会更早碰到结果文件格式、命名规范、版本兼容等稳定性问题
- 增加本轮设计复杂度

### 5.3 方案 C：直接做通用 Runner

一次性把 front1 / front2 / CRA40 / ERA5 / 多变量全部抽象进统一入口。

优点：

- 形式上最完整

缺点：

- 当前阶段风险最高
- 会把尚未统一验证的链路一起耦合进来
- 设计复杂度和误判概率都明显上升

### 5.4 采用方案

本次采用方案 A。

理由很明确：当前最重要的不是追求“架子搭得最大”，而是把已经通过真实数据验证的能力，以清楚、可靠、可维护的方式正式固化下来。

## 6. 技术架构与旧工程映射文档设计

### 6.1 文档定位

该文档面向维护者、后续扩展者和需要回查 legacy 研究逻辑的人。

它与现有文档的关系应为：

- 不替代快速使用指南
- 不替代分层设计文档
- 不替代 legacy 冒烟基线文档
- 它负责把“当前模块边界”和“旧新逻辑承接关系”说清楚

### 6.2 推荐文件名

建议文件名：

- `docs/pipeline_architecture_mapping_zh.md`

### 6.3 推荐结构

推荐章节如下：

1. 文档目的与读者
2. 当前新流水线总体结构
3. 模块边界与职责
4. 新模块与旧脚本映射表
5. 已迁移能力 / 未迁移能力
6. 后续扩展接入建议
7. 与现有文档的使用顺序

### 6.4 映射原则

映射文档必须坚持以下原则：

- 按研究流程和模块职责映射，而不是按旧脚本文件名机械拆分
- 一个旧脚本可以映射到多个新模块
- 一个新模块也可以承接多个旧脚本的公共逻辑
- 明确“当前只迁移了接口与公共计算框架”还是“已经迁移了完整科研生产能力”

### 6.5 当前建议映射

当前建议在文档中明确以下对应关系：

- `inventory`
  - 对应：当前整理出的旧工程运行基线与环境依赖检查
- `masks`
  - 对应：`front_grid_lon_lat_unification.py`
  - 以及 `frontal_processing_CRA40.py` 中与 offset 掩膜输出相关的部分
- `geometry`
  - 对应：`frontal1_process_w.py`
  - `frontal1_process_rh.py`
  - `frontal2_process.py`
  - `frontal1_process_SelectSubArea.py`
  - 其中主要承接“中心线拟合、切线/法线采样框架”
- `profiles`
  - 对应：`frontal1_process_w.py`
  - `frontal1_process_rh.py`
  - `frontal2_process.py`
  - 其中主要承接“沿切线抽取 RH / 温度 / 垂直速度等剖面”
- `subareas`
  - 对应：`frontal1_process_SelectSubArea.py`
- `statistics`
  - 对应：`merge_csv.py`
  - `merge_csv fengmian2.py`
- `diagnostics`
  - 当前仍主要保留在 legacy 侧
  - 主要对应：`frontal_info_graphic_identification.py`
  - `frontal_processing_CRA40.py` 中的诊断图部分

### 6.6 迁移状态表达

文档中必须明确区分三类状态：

- 已迁移并已验证
- 已有模块边界，但仍主要依赖 legacy 输出
- 尚未迁移到新流水线总入口

这样后续扩展者才能知道“能直接复用什么”和“还不能假定什么”。

## 7. Runner 升级设计

### 7.1 升级目标

将 `runner` 从“基础入口”升级为“当前最小已验证链路的串联入口”。

该入口应串联以下步骤：

1. `inventory`
2. `masks`
3. 主掩膜读取
4. `geometry`
5. RH 变量场读取与网格对齐
6. `profiles`
7. `subareas`
8. 完整锋面均值统计
9. 子区域均值统计

### 7.2 设计边界

升级后的 `runner` 只支持当前已验证链路：

- `cfg.dataset == "cra40"`
- `cfg.front_id == "front2"`
- `cfg.target_time == "2017-06-22T18"` 或至少仍建立在已存在的 `CRA40 front2` 掩膜资产约束上

如果调用超出该边界的配置，应明确报出“当前 runner 仅正式支持已验证链路”，而不是静默尝试泛化。

### 7.3 输入来源

升级后的 `runner` 应主要消费：

- `pipeline/schemas/pipeline_config.yaml`
- `project_paths.py` 中的项目内路径体系
- `nc_compat.py` 提供的兼容读取入口
- `pipeline.steps.*` 已有模块接口

### 7.4 输出形态

升级后的 `runner` 暂不要求完整落盘全部中间结果，优先返回一个结构化摘要字典，包含：

- `case_name`
- `inventory`
- `masks`
- `outputs`
- `geometry`
- `profiles`
- `subareas`
- `statistics`

其中各部分至少提供足以判断执行是否正常的摘要信息，例如：

- `geometry`
  - `centerline_points`
  - `section_shape`
- `profiles`
  - `variable`
  - `bundle_shape`
- `subareas`
  - `mask_shape`
  - `selected_points`
- `statistics`
  - `front_mean`
  - `subarea_mean`

### 7.5 为什么先返回摘要而不是全量文件

原因有三个：

1. 当前最需要的是统一串联和可验证性
2. 全量文件落盘格式尚未完全稳定
3. 摘要输出更适合作为后续继续扩展的稳定壳层

后续如果要增加 manifest 或更完整落盘，可以在当前摘要结构之上演进，而不必重写入口边界。

### 7.6 变量策略

本轮 `runner` 中，`profiles` 与 `statistics` 先聚焦 `rh`，因为：

- 这是当前真实案例中已完成验证的变量
- 它已经具备真实参考值和真实形状基线

后续如要扩展 `temp / w / thetae`，建议沿着相同接口逐步接入，而不是在本轮一次性全部绑定。

### 7.7 子区域策略

当前 `runner` 中的子区域步骤先采用一个固定、已验证的案例参数：

- `start_section=1`
- `end_section=4`

并在摘要中明确写出这些参数，避免用户误以为它是通用默认规则。

### 7.8 错误处理策略

升级后的 `runner` 需要对以下情况给出清晰报错：

- 配置与当前已验证链路不匹配
- 所需掩膜文件缺失
- RH 原始场文件缺失
- 主掩膜与变量场对齐失败
- 子区域掩膜读取或对齐失败

报错应优先解释“当前支持边界”与“缺失的输入是什么”，而不是只抛底层异常。

## 8. 测试与验证设计

### 8.1 文档侧验证

技术架构与旧工程映射文档的验证标准为：

- 新模块职责边界表述清晰
- 旧脚本映射关系可追溯
- 已迁移/未迁移边界明确
- 与现有设计文档、快速使用指南不冲突

### 8.2 Runner 侧验证

`runner` 升级完成后，至少需要验证：

- 现有 `test_runner_step.py` 随升级同步更新
- `run_case(...)` 返回更多步骤摘要
- 真实案例 `CRA40 front2 2017-06-22T18` 能成功串联
- 返回值中能看到几何、剖面、子区域、统计四类摘要

### 8.3 真实案例参考基线

当前真实案例的参考基线包括：

- `geometry.centerline_x.shape == (8,)`
- `geometry.sampled_dx.shape == (8, 9)`
- `bundle.values.shape == (8, 9, 37)`
- `submask.shape == (81, 141)`
- 完整锋面均值约 `85.81288`
- 子区域均值约 `79.697523`

这些值不一定都要在测试里逐个硬编码为高精度断言，但应作为 runner 摘要验证与人工冒烟对照的参考基线。

## 9. 成功标准

如果本次设计落实成功，应达到以下效果：

- 维护者可以通过一份文档理解新模块边界与旧工程承接关系
- 使用者或维护者可以通过升级后的 `runner` 统一跑通当前已验证链路
- 新流水线的“可维护性”和“可调用性”都明显高于当前仅靠手工拼接的状态
- 本轮没有过早泛化到尚未稳定验证的 front1 / ERA5 / 多变量总控

## 10. 下一阶段接口

本轮完成后，下一阶段才适合继续推进：

- 更完整的多变量 profile 串联
- front1 接入
- ERA5 接入
- 诊断图入口模块化
- 结果文件落盘格式规范化

也就是说，本轮的角色不是“最终形态”，而是“把当前最可信的一条链路封装成稳定壳层，并把新旧关系写明白”。
