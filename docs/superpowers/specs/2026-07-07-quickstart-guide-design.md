# 梅雨锋新流水线快速使用指南设计文档

## 1. 目标

本设计文档用于定义一份面向使用者的“快速使用指南”，帮助研究者在不需要通读全部新旧代码的前提下，能够理解并实际运行当前已经落地的新流水线能力。

这份指南的首要目标不是解释全部架构细节，而是回答以下问题：

- 当前这套新流水线已经能做什么
- 运行前需要准备什么
- 从哪里开始执行
- 每一步输入和输出是什么
- 常见报错该如何定位
- 结果文件通常会落到哪里

## 2. 读者与适用范围

目标读者包括：

- 当前项目的主要研究者
- 未来需要复现实验或接手维护的人
- 对旧工程全貌不完全熟悉，但需要使用新流水线结果的人

这份指南默认读者：

- 知道项目总体研究目标
- 知道 CRA40、ERA5、锋面掩膜、剖面、子区域、统计序列这些基本概念
- 不要求记住旧工程每个脚本的细节

这份指南不负责：

- 展开全部科研判据与气象业务背景
- 逐行解释所有新模块内部算法
- 替代后续的“技术架构与旧工程映射”文档

## 3. 当前需要解决的使用问题

结合当前项目状态，使用层面最突出的困难有三类：

### 3.1 入口不清晰

虽然 `pipeline/` 已经具备 `masks / geometry / profiles / subareas / statistics` 这些模块，但对于使用者来说，“从哪一步开始”“哪些步骤已经连通”“哪些步骤还需要手动拼接”还不够直观。

### 3.2 数据与网格关系不直观

当前流程中存在多套网格与掩膜：

- 主锋面掩膜
- extend 掩膜
- 子区域掩膜
- CRA40 / ERA5 变量场

使用时最容易犯错的地方，是把不同网格上的场和掩膜直接混用，导致统计结果错误或采样失败。

### 3.3 Windows 中文路径兼容问题不容易预判

本项目位于中文目录下，旧工程与新流水线都已经实际遇到过：

- `xarray/netCDF4/cfgrib/pygrib` 对中文绝对路径兼容不稳定
- PowerShell 终端中文显示与程序实际读写行为不一致

因此快速指南必须把这些兼容约束明确写出来，而不能默认使用者自然知道。

## 4. 推荐文档形态

推荐新增一份独立文档，定位为“直接上手说明”，而不是设计说明或开发说明。

建议文件名采用：

- `docs/pipeline_quickstart_zh.md`

原因：

- 放在 `docs/` 根下，便于和现有设计文档并列查看
- 文件名直接表达用途，适合作为后续入口文档
- 与未来的维护文档、映射文档形成平行关系

## 5. 文档结构设计

快速使用指南采用六个部分，按“先能跑，再理解，再排错”的顺序组织。

### 5.1 这份指南适合谁

说明指南的使用对象、阅读前提、当前版本定位。

必须明确：

- 这不是完整科研算法说明书
- 这是一份“当前版本如何使用”的实操说明
- 当前流水线仍处于“模块已打通、总入口待继续串联”的阶段

### 5.2 运行前准备

说明运行环境、目录约定和必要数据。

必须覆盖：

- 虚拟环境：`cwr_py312`
- 项目根目录位置
- 推荐使用的项目内数据目录
- 运行前要确认的关键数据资产
- 中文路径兼容提醒

这里不要求罗列全部原始资料清单，但必须点出最常见、最关键的输入类别，例如：

- 原始 CRA40 / ERA5 资料
- 主掩膜、extend 掩膜、offset 掩膜、子区域掩膜
- 已存在的 CSV 统计结果或待统计的网格场数据

### 5.3 当前流水线已经具备的能力

说明当前新流水线并不是“全自动一条命令到底”，而是已经形成了可复用的研究步骤骨架。

必须清楚交代：

- 当前已具备的主链路为：`mask -> geometry -> profiles -> subareas -> statistics`
- `runner.py` 当前仍是基础入口，不等于完整科研总调度器
- 使用者目前更适合按步骤执行和验证，而不是默认期待一步全跑完

### 5.4 最小可运行示例

这是整份指南的核心部分。

建议使用当前已经完成真实冒烟验证的个例：

- `CRA40 front2 2017-06-22T18`

示例应按顺序展示：

1. 读配置或明确案例参数
2. 解析掩膜
3. 从主掩膜构建几何结果
4. 基于真实变量场抽取剖面
5. 依据 sections 生成子区域掩膜
6. 对完整锋面区或子区域做掩膜平均统计

每一步都必须写清：

- 输入是什么
- 调用哪个模块或函数
- 输出对象长什么样
- 使用者应如何判断这一步是否正常

示例优先采用当前项目内已经验证过的真实函数，而不是重新发明包装接口。

### 5.5 常见问题

必须收录当前最可能复发的三类问题：

- 路径错误或找不到数据
- 掩膜与变量场不在同一网格
- Windows / PowerShell 中文路径与编码兼容问题

每类问题都要尽量用“症状 -> 原因 -> 处理方式”的格式写明，避免只有原则没有操作建议。

### 5.6 结果文件与后续定位

说明输出通常去哪里，以及如何和旧工程结果对应。

必须覆盖：

- `outputs/` 下当前新流水线的典型目录
- 使用过程中的中间结果是内存对象还是文件
- 哪些结果仍主要由 legacy 脚本输出
- 如果要回查旧工程基线，应看哪几份现有文档

## 6. 内容边界与写作原则

这份快速使用指南必须遵守以下边界：

### 6.1 以使用为中心

优先回答“怎么用”，而不是优先解释“为什么这么设计”。

### 6.2 只写当前已验证能力

凡是还没有接入总入口、还没有经过真实数据冒烟验证的能力，不应写成“已经完全可用”的口吻。

### 6.3 用项目内真实路径与真实案例

尽量使用当前项目目录中的真实位置、真实文件类型和真实案例时间，避免写成抽象示意文档。

### 6.4 中文注释与术语保持一致

文档中的术语应尽量与当前项目中已经形成的叫法一致，例如：

- 主掩膜
- extend 掩膜
- 子区域掩膜
- 几何结果
- 剖面 bundle
- 掩膜平均统计

这样后续再写维护文档时，新旧术语才能自然对应。

## 9. 成功标准

如果这份快速使用指南达到以下标准，就说明它是成功的：

- 不熟悉新流水线内部结构的人，能知道从哪里开始
- 使用者能照着最小示例把一条真实链路跑下来
- 使用者知道当前哪些能力已经落地，哪些还不是“一键总流程”
- 使用者遇到路径、网格、中文路径兼容问题时，能快速定位原因
- 这份文档可以作为后续“技术架构与旧工程映射”文档的前置入口
